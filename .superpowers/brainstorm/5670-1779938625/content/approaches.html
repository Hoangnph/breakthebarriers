3 Hướng Tiếp Cận

Chọn kiến trúc phù hợp nhất với mục tiêu của bạn

A

asyncio Pools + PostgreSQL Queue

Nâng cấp kiến trúc hiện tại, không thêm infrastructure

Thêm vào
• Bảng jobs trong PostgreSQL
• asyncio.Semaphore theo volume tier
• SSE endpoint cho progress
• Volume Detector service
Giữ lại
• FastAPI + uvicorn
• BackgroundTasks pattern
• PostgreSQL duy nhất
• Không cần Redis/broker

Pros

  • Không ops mới
  • Deploy đơn giản
  • Đủ cho ~20 users đồng thời

Cons

  • Bound to 1 process
  • Không horizontal scale
  • Queue logic tự viết
B

Celery + Redis — Full Production Queue

Job queue phân tán thực sự, enterprise-grade

Thêm vào
• Redis (broker + cache)
• Celery workers theo tier
• Flower dashboard (monitoring)
• WebSocket progress
Capabilities
• Horizontal scaling worker
• Priority queues per tier
• Built-in retry + dead-letter
• Resume từ checkpoint

Pros

  • Scale không giới hạn
  • Battle-tested production
  • Monitoring đầy đủ

Cons

  • Thêm Redis infra
  • Phức tạp hơn deploy
  • Over-engineering nếu <50 users
C

⭐ Hybrid Adaptive — Khuyến nghị

asyncio cho S/M, Celery+Redis chỉ khi cần cho L/XL

S / M docs
→ asyncio pool
→ SSE progress
→ Response nhanh
Không cần Redis
⟷
L / XL docs
→ Celery queue
→ Priority lanes
→ Resume + checkpoint
Redis chỉ khi cần
Volume Detector (module mới)
📏 Đếm trang PDF 📊 Ước tính tokens 💰 Estimate cost 🎯 Chọn quality tier 🔀 Route tới đúng pipeline

Pros

  • Không over-engineer S/M
  • Scale khi cần L/XL
  • Cost-efficient
  • Progressive rollout

Cons

  • 2 code paths cần maintain
  • Redis vẫn cần khi có L/XL
  • Volume Detector phải chính xác
